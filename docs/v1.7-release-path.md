# v1.7 Release Path

This document defines the narrow deployment-and-operator-maturity release path for ClarityOS `v1.7` and the release gates required to call that path measurably better than `v1.6`.

## First Packaged Deployment Use Case

ClarityOS `v1.7` is aimed at a narrow but stronger self-hosted deployment shape:

- single-tenant
- self-hosted
- browser-first and API-first
- packaged rather than repo-habit-dependent
- operator-visible
- persisted-state-aware
- supportable rather than bespoke

In concrete terms, the first supported `v1.7` deployment path is:

`A self-hosted assistant and operator runtime that can be started through one documented packaged profile, preserves explicit persisted state across restarts, and gives operators enough visibility and playbook support to run it credibly without turning into a hosted multi-tenant platform.`

## Supported Shape

The `v1.7` release path supports:

- the web-first assistant at `/assistant`
- the embeddable widget at `/widget` and `/widget.js`
- the operator console at `/operator`
- one documented packaged startup path for the supported self-hosted profile
- one explicit state-root contract for packaged deployment, including `CLARITYOS_STATE_ROOT`
- explicit persisted-state directories and deployment-safe environment expectations
- operator-visible health, maintenance, and recovery posture suitable for repeated self-hosted operation

## Deliberately Out Of Scope

The `v1.7` release path does not claim support for:

- hosted hub infrastructure
- broad SaaS-style multi-tenancy
- cluster orchestration breadth for every environment
- automatic cloud provisioning across many providers
- marketplace-style plugin hosting or remote control planes

`v1.7` is the packaged self-hosted deployment release, not the hosted platform release.

## Release Gates

ClarityOS `v1.7` is ready for this packaged deployment path only if all of the following are true.

### Gate 1: Validation Passes

- `scripts/run_release_validation.sh --full` passes cleanly.
- `python -m unittest discover -s tests -v` passes cleanly.
- the packaged startup path is documented clearly enough to reproduce from a fresh environment.

### Gate 2: Startup And Runtime Profile Stay Explicit

- the supported startup path does not depend on undocumented local shell habits
- the runtime profile for assistant, widget, operator, and background execution is explicit
- packaged deployment assumptions are documented rather than inferred

### Gate 3: Storage And Environment Posture Stay Credible

- the packaged deployment path uses one explicit state-root contract rather than many undocumented mount assumptions
- persisted state directories and backup expectations are documented clearly
- operators can distinguish what must be preserved from what can be regenerated
- environment and filesystem assumptions are narrow enough to support repeatable self-hosted use

### Gate 4: Operator Supportability Is Credible

- operators can inspect health, maintenance posture, and recovery state without guessing
- the operator dashboard exposes a clear runtime-posture summary with recommended next actions for queue, worker, storage, and session drift
- deployment-safe maintenance and incident guidance exist for the supported profile
- repeated self-hosted operation feels more supportable than the `v1.6` baseline

### Gate 5: The Supported Use Case Is Believable

Operators can credibly use `v1.7` for:

- standing up a repeatable local or team-owned ClarityOS deployment
- preserving runtime state across restarts and maintenance windows
- onboarding operators without relying on private setup knowledge
- running the supported browser-first assistant profile with clearer deployment boundaries

### Gate 6: Scope Remains Narrow

The release must stay within the declared `v1.7` support boundary:

- better self-hosted repeatability, not hosted platform sprawl
- explicit deployment posture, not environment-by-environment improvisation
- stronger operator supportability, not cloud-control-plane expansion
- one believable packaged deployment path, not infrastructure breadth for its own sake

If a candidate depends on broad hosted infrastructure or cluster-management scope, it belongs after `v1.7`.

## Done Means Repeatable Self-Hosting, Not Platform Expansion

For `v1.7`, “done” does not mean ClarityOS becomes a hosted assistant platform.

It means:

- one packaged startup path is documented and supportable
- persisted state and backup expectations are clearer
- operators can run the supported profile with less local folklore
- one believable self-hosted deployment path is measurably better than `v1.6`
