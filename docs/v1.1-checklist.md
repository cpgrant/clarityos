# ClarityClaw v1.1 Checklist

Concrete implementation checklist for `v1.1`, the first assistant surface and session gateway milestone.

Status: completed

## Slice 1: Session And Conversation Model

- [x] `runtime/session.py`
  Add explicit session or conversation records that map user interactions to workflow ids, run ids, memory scope, and operator-visible metadata.
- [x] `runtime/session.py`
  Define session lifecycle rules for open, active, waiting, errored, archived, and recovered conversations.
- [x] `api/main.py`
  Add session-oriented endpoints for create, inspect, list, and append-message flows without bypassing the existing workflow runtime.
- [x] `tests/test_session.py`, `tests/test_api.py`, `tests/test_control_plane.py`
  Cover conversation-to-workflow mapping, memory continuity, session recovery, and operator inspection.

## Slice 2: First Assistant Surface

- [x] `api/` or new `gateway/`
  Add a web-first assistant surface that sends inbound messages into the existing workflow and queue layers.
- [x] `README.md`
  Document the first supported assistant use case, startup flow, and scope boundaries.
- [x] `tests/`
  Cover end-to-end request/response flows for the first assistant surface, including async and recovery cases.

## Slice 3: Operator UI And Conversation Inspection

- [x] `ui/` or equivalent
  Add a lightweight operator-facing UI for sessions, workflows, incidents, queue health, and worker health.
- [x] `runtime/control_plane.py`
  Expose any additional session-oriented summaries needed by the operator UI without duplicating existing workflow or incident logic.
- [x] `README.md`
  Document how operators inspect, recover, and trace a live conversation through the UI and control plane.

## Slice 4: First External Channel Or Gateway Adapter

- [x] `gateway/` or equivalent
  Add one thin external channel adapter only after the web-first assistant surface is stable.
- [x] `runtime/` / `api/main.py`
  Keep channel adapters transport-focused so they normalize inbound messages and hand off to the existing session/workflow model.
- [x] `docs/roadmap.md`
  Record which first external channel is chosen and why it is narrow enough to support well.
- [x] `README.md`
  Clarify what remains intentionally out of scope so `v1.1` stays focused on one strong use case rather than channel sprawl.
