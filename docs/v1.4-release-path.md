# v1.4 Release Path

This document defines the narrow tool-maturity release path for ClarityClaw `v1.4` and the release gates required to call that path measurably better than `v1.3`.

## First Tool-Maturity Use Case

ClarityClaw `v1.4` is aimed at a narrower but more capable assistant/runtime shape:

- single-tenant
- self-hosted
- browser-first and API-first
- session-backed
- operator-visible
- retrieval-stronger
- maintenance-capable

In concrete terms, the first supported `v1.4` tool-maturity path is:

`A self-hosted assistant and operator runtime that can inspect repo, runtime, and bounded external sources more usefully, and can perform narrow operator-safe maintenance actions through an explicit maintenance agent without broad write or shell execution.`

## Supported Shape

The `v1.4` release path supports:

- the web-first assistant at `/assistant`
- the embeddable widget at `/widget` and `/widget.js`
- the operator console at `/operator`
- stronger bounded retrieval through:
  - repo search and file excerpts
  - bounded directory navigation
  - compact runtime inspection summaries
  - controlled external fetches on configured domains
- clearer retrieval outputs that include previews, counts, summaries, and matched-file metadata
- explicit maintenance actions through `maintenance_operator` for:
  - session archival and session pruning
  - queue promotion and stale-job repair
  - orphaned-worker repair
  - safe workflow resume, replay, and recovery

## Deliberately Out Of Scope

The `v1.4` release path does not claim support for:

- broad repo write helpers
- arbitrary shell or exec-like mutation
- unrestricted operator mutation from general assistant agents
- many-channel expansion
- plugin marketplace behavior
- hosted multi-tenant operation

`v1.4` is the tool-maturity release, not the broad action-platform release.

## Release Gates

ClarityClaw `v1.4` is ready for this tool-maturity path only if all of the following are true.

### Gate 1: Validation Passes

- `scripts/run_release_validation.sh` passes cleanly.
- `python -m unittest discover -s tests -v` passes cleanly.
- `python -m unittest tests.test_agent tests.test_policy -v` passes cleanly.

### Gate 2: Tool Registry Structure Stays Explicit

- `runtime.tools` remains the single explicit registry facade
- tool handlers are organized by domain without introducing hidden routing behavior
- capability metadata, path/domain handling, and tool lookup stay visible at the registry boundary

### Gate 3: Retrieval Is More Useful

- repo tools return enough metadata for downstream answer shaping, not just raw payloads
- runtime inspection tools provide compact summaries as well as structured detail
- controlled external fetches stay within configured domains and return compact summaries
- tool-guided grounding can reuse the stronger retrieval outputs directly

### Gate 4: Action Scope Stays Narrow

- maintenance actions are routed through an explicit `runtime_write` capability
- maintenance actions are available only to the dedicated maintenance agent profile
- non-maintenance agents are denied when they attempt maintenance tools
- the shipped action set remains runtime-maintenance-only

### Gate 5: Operator Maintenance Is Believable

Operators can credibly use `v1.4` for:

- repo and runtime research with stronger retrieval context
- queue and worker inspection with clearer summaries
- archiving and pruning session state intentionally
- promoting due jobs and repairing stale queue state
- repairing orphaned workers
- safely resuming, replaying, and recovering workflows through explicit bounded actions

### Gate 6: Scope Remains Narrow

The release must stay within the declared `v1.4` support boundary:

- stronger tools, not broader magic
- runtime maintenance, not broad arbitrary mutation
- bounded retrieval and action, not shell-like execution
- explicit operator posture, not hidden autonomy

If a candidate depends on repo write helpers or broad execution, it belongs after `v1.4`.

## Done Means More Capable, Not Broader

For `v1.4`, “done” does not mean ClarityClaw becomes a full coding agent platform.

It means:

- the tool layer is cleaner and easier to extend
- retrieval results are more useful to both humans and the assistant
- bounded maintenance actions exist for real operator tasks
- action power remains explicit, narrow, and policy-scoped
- one believable tool-maturity path is measurably better than `v1.3`

That is the line between “better grounded assistant/runtime” and “broader mutation platform.”
