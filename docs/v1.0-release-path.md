# v1.0 Release Path

This document defines the first production use case for ClarityClaw `v1.0` and the gates required to call the runtime ready for that use case.

## First Production Use Case

ClarityClaw `v1.0` is aimed at a narrow, operator-managed deployment:

- single-tenant
- API-first
- self-hosted
- async workflow execution
- bounded agent behavior
- operator-visible recovery and maintenance

In concrete terms, the first supported production path is:

`An internal workflow runtime for durable, operator-visible assistant and research tasks started through the API and executed through the queue/worker system.`

## Supported Shape

The `v1.0` release path supports:

- `POST /workflows` and `POST /jobs` as the primary entry points
- bounded default and researcher-style agents
- dedicated memory operations through the memory operator
- queue-backed execution with worker claiming, retry, replay, and safe resume
- operator inspection through workflow, incident, queue-health, worker-health, and state endpoints
- single-tenant operational ownership with explicit tokens and front-proxy TLS

## Deliberately Out Of Scope

The `v1.0` release path does not claim support for:

- end-user chat UI
- multi-channel messaging surfaces
- hosted multi-tenant operation
- plugin marketplace behavior
- broad arbitrary tool execution in production
- high-trust autonomous delegation by default

Those are future concerns. `v1.0` is the trusted-runtime release, not the assistant-surface release.

## Release Gates

ClarityClaw `v1.0` is ready for this production path only if all of the following are true.

### Gate 1: Validation Passes

- `scripts/run_release_validation.sh` passes cleanly.
- `python -m unittest discover -s tests -v` passes cleanly.

### Gate 2: Production Posture Is Explicit

- `CLARITYCLAW_ENV=production` is set in the deployed environment.
- `CLARITYCLAW_OPERATOR_TOKEN` is set.
- the deployed config files are explicitly selected, ideally with:
  - `CLARITYCLAW_AGENTS_CONFIG=config/agents.production.yaml`
  - `CLARITYCLAW_POLICIES_CONFIG=config/policies.production.yaml`

### Gate 3: Operator Profile Matches Intent

- `GET /operator/profile` reports `production_mode: true`
- operator auth is enabled
- config paths point to the intended runtime profile

### Gate 4: Health Starts Clean

Before release, a clean runtime boot should show:

- no expired running jobs in `GET /queue/health`
- no orphaned workers in `GET /workers/health`
- no unexpected incident blockers in a representative workflow drill

### Gate 5: Recovery Flows Are Runnable

Operators can successfully run the documented playbooks for:

- migration
- queue cleanup
- worker repair
- workflow recovery
- incident response

This does not require every flow to be exercised in a live outage, but it does require that the operator procedures are concrete, current, and mapped to real endpoints.

### Gate 6: Scope Remains Narrow

The release must stay within the declared `v1.0` support boundary:

- API-first, not UI-first
- self-hosted, not hosted SaaS
- single-tenant, not general multi-tenant
- bounded runtime behavior, not open-ended autonomous expansion

If a release candidate depends on broader product surface area, it belongs after `v1.0`.

## Done Means Shipped

For `v1.0`, “done” does not mean every future use case is unlocked.

It means:

- the runtime can be deployed with explicit production posture
- operators can inspect and recover it safely
- the bounded async workflow path is trustworthy under failure and restart
- the supported use case is narrow enough that reliability claims are believable

That is the line between “hardened project” and “trusted runtime.”
