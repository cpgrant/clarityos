# ClarityOS v1.7 Checklist

Status: in progress

Goal:
Make ClarityOS easier to package, deploy, and operate repeatedly as a self-hosted assistant system.

Companion docs:

- [`docs/roadmap.md`](./roadmap.md)
- [`docs/differentiators.md`](./differentiators.md)
- [`docs/v1.7-release-path.md`](./v1.7-release-path.md)

## Slice 1: Packaging And Runtime-Profile Baseline

Status: complete

Focus:

- make the supported startup path repeatable instead of dependent on local repo habits
- define the first clear packaged runtime profile for self-hosted use

Candidate work:

- add a first Podman or container-oriented packaging baseline
- define the supported runtime profile for assistant, widget, operator, and background execution
- document a first clean-environment startup path that does not depend on implicit local knowledge

Completed through:

- a first `Containerfile` baseline for packaged API startup
- `compose.yaml` wiring a packaged API service and packaged worker service over shared persisted-state mounts
- `runtime.worker_loop` as the first repeatable background job runner for packaged deployments
- README guidance for Podman Compose or Docker Compose startup and the supported packaged runtime profile

Acceptance cues:

- a fresh environment can start the supported profile through one documented path
- runtime posture is explicit rather than inferred from shell history and repo state

## Slice 2: Storage Layout, Backup, And Deployment-Safe Environment Rules

Status: complete

Focus:

- make persisted state easier to preserve, migrate, and reason about in deployed setups
- reduce ambiguity around what must be backed up, mounted, or can be regenerated

Candidate work:

- clarify storage layout expectations for sessions, memories, artifacts, logs, and runtime state
- add backup and restore guidance or helpers for supported state directories
- define environment and filesystem assumptions that stay safe in a packaged deployment

Completed through:

- an explicit `CLARITYOS_STATE_ROOT` contract for packaged deployments
- a single packaged state mount in `compose.yaml` instead of many separate implicit directory mounts
- operator-profile storage metadata covering backup priority, preserve-on-restart posture, and regeneration expectations
- a storage and backup playbook for self-hosted operators

Acceptance cues:

- operators can explain what data must be preserved and how to restore it
- deployed storage expectations are clearer than the current repo-root default posture

## Slice 3: Operator Ergonomics And Observability For Repeated Operation

Status: pending

Focus:

- make repeated day-to-day operation less manual and more supportable
- improve operator trust in health, maintenance, and recovery views for packaged deployments

Candidate work:

- improve operator summaries for runtime health, storage posture, and maintenance state
- add clearer observability hooks, counters, or operator-facing status views
- tighten playbooks for deployment-safe maintenance and incident response

Acceptance cues:

- operators can assess runtime posture more quickly from the control surface
- self-hosted operation feels more repeatable and supportable than the `v1.6` baseline

## Slice 4: Release Path For Repeatable Self-Hosted Deployment

Status: pending

Focus:

- define the first narrow supportable packaged deployment path for `v1.7`

Candidate work:

- document the supported deployment profile improved by `v1.7`
- define release gates for packaging, storage posture, and operator supportability
- define what remains intentionally out of scope

Acceptance cues:

- `v1.7` can be described as deployment and operator maturity rather than just more docs
- release status can be justified by repeatability and supportability improvements
