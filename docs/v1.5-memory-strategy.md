# ClarityClaw v1.5 Memory Strategy

This document is a companion to [`docs/v1.5-checklist.md`](./v1.5-checklist.md).

Its job is to clarify the product and design intent behind `v1.5` memory and continuity work without replacing the execution checklist.

## Why This Exists

`v1.5` is not just about storing more context. The goal is to make longer-running assistant sessions more coherent while preserving ClarityClaw's existing strengths:

- explicit persisted state
- inspectable memory records
- bounded prompt construction
- operator-visible runtime behavior

The desired outcome is continuity with traceability, not hidden long-context dumping.

## Design Principles

- Keep raw history. Summaries should help carry context forward, not erase what happened.
- Make summaries explicit. A continuity summary should be a first-class persisted record or state object, not an invisible prompt-side transformation.
- Preserve attribution. Summaries should point back to source messages, memory ids, workflow ids, or time windows.
- Stay bounded. Carry-forward context should have clear size, count, and scope limits.
- Separate memory from automation. Remembering a user preference is not the same thing as taking action automatically.
- Keep operators in the loop. Continuity decisions should be visible through control-plane views rather than hidden inside model prompts.

## Working Model

The likely `v1.5` memory shape is:

1. raw session messages remain persisted in `sessions/`
2. typed memory records remain persisted in `memories/`
3. continuity summaries are created explicitly for bounded carry-forward
4. operator views expose current continuity state, compaction status, and source references

In this model, the assistant should carry forward:

- a bounded continuity summary
- a bounded tail of recent messages
- a bounded set of still-relevant raw memories

It should not carry forward an ever-growing opaque transcript.

## Slice 1 Guidance

Slice 1 should stay narrow and execution-focused.

Recommended direction:

- add explicit summarization helpers for older session and workflow context
- create inspectable summary outputs with source references
- add bounded compaction rules instead of silent replacement
- keep raw records available for audit and debugging

Helpful metadata for summary or compaction outputs:

- covered message ids
- source memory ids
- related workflow ids
- summary purpose
- compaction timestamp
- summary version or strategy id

## What Not To Do

Avoid these traps in `v1.5`:

- replacing old context with one rolling prose blob
- inferring hidden user profiles from weak signals
- silently mutating prompts without persisted evidence
- coupling continuity too early to proactive automation
- letting carry-forward become an unbounded background dump

## Preferences And Longer-Term Help

Preference memory is promising, but it should be treated as a later layer on top of continuity, not as the first deliverable of Slice 1.

Good future directions:

- explicit user preferences such as tone, formatting, or recurring workflow habits
- operator-visible preference records
- opt-in recurring helper behaviors with clear boundaries

Not for early `v1.5`:

- broad autonomous action based on inferred behavior
- silent background task execution
- large-scale behavioral profiling
