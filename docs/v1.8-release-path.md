# v1.8 Release Path

This document defines the narrow email-triage release path for ClarityClaw `v1.8` and the release gates required to call that path measurably better than the `v1.7` baseline.

## First External Integration Use Case

ClarityClaw `v1.8` is aimed at one careful external integration shape:

- single-tenant
- self-hosted
- browser-first and API-first
- operator-visible
- approval-aware
- email-triage-focused rather than mailbox-management-focused

In concrete terms, the first supported `v1.8` path is:

`A self-hosted assistant runtime that can ingest a bounded email payload, produce explicit triage state and a draft-reply suggestion, and route any draft-approval step through visible operator controls without claiming autonomous inbox management.`

## Operator Walkthrough

One believable `v1.8` operator flow now looks like this:

1. An operator submits a bounded email payload to `/intake/email`.
2. ClarityClaw creates or reuses a normal session, runs the email through the existing workflow runtime, and persists a first-class `email_triage` artifact.
3. The operator opens the workflow or session in the operator console and reviews the structured triage fields such as bottom line, urgency, suggested bucket, next action, and draft reply.
4. If the draft suggestion looks worth reviewing, the operator triggers `Request Draft Approval`, which creates a visible approval record tied to the triage artifact.
5. The operator explicitly approves, denies, or aborts that draft review from the same operator surface.
6. If the draft is approved, the operator can materialize an `email_draft_handoff` artifact that marks the draft as ready for manual follow-through outside ClarityClaw.
7. ClarityClaw still stops at the handoff boundary: it does not send the email, mutate the mailbox, or imply autonomous outbound behavior in `v1.8`.

This walkthrough is intentionally narrow. It demonstrates a useful external-work path without weakening the runtime's inspectability or approval posture.

## Supported Shape

The `v1.8` release path supports:

- operator-authenticated email intake through `/intake/email`
- one-email or one-thread-style bounded triage input rather than broad mailbox sync
- explicit source metadata such as account, thread, message, sender, and timestamps
- structured email triage output with fields such as bottom line, urgency, suggested bucket, recommended next action, and optional draft reply
- persisted `email_triage` artifacts tied to the normal workflow and session runtime
- explicit approval requests for draft replies through `/artifacts/{artifact_id}/email-draft-approval`
- explicit approved-draft handoff artifacts through `/artifacts/{artifact_id}/email-approved-draft-handoff`
- operator visibility for triage state and draft-approval status through the control plane and operator UI
- one operator walkthrough from intake to approved handoff that stays inside the bounded runtime

## Deliberately Out Of Scope

The `v1.8` release path does not claim support for:

- autonomous email sending
- mailbox mutation such as archive, label, delete, move, or reply-send
- full inbox synchronization or background mailbox mirroring
- broad multi-channel parity with OpenClaw-style chat surface breadth
- hidden preference learning or silent inbox automation

`v1.8` is the careful external email-triage release, not the autonomous mailbox release.

## Release Gates

ClarityClaw `v1.8` is ready for this narrow email-triage path only if all of the following are true.

### Gate 1: Validation Passes

- `python -m unittest discover -s tests -v` passes cleanly.
- email-intake, control-plane, and browser-smoke coverage includes the bounded intake and approval path.
- release validation includes a bounded synthetic drill from email intake to structured triage to draft approval to approved handoff.
- the supported email-triage flow is documented clearly enough to reproduce from a fresh environment.

### Gate 2: Intake Boundaries Stay Explicit

- email intake happens through one documented bounded adapter path
- supported payload shape and size limits remain explicit
- email-origin state is visible afterward in sessions, workflows, artifacts, and operator views
- the integration still feels like a thin adapter into the runtime rather than a mailbox subsystem

### Gate 3: Triage Output Is Useful And Inspectable

- structured triage output is persisted with stable fields rather than only a raw assistant reply
- a user can hand ClarityClaw one bounded email item and receive a useful triage result
- operator views can surface triage output without custom log-diving or hidden prompt state

### Gate 4: Approval Posture Is Credible

- any draft-reply approval request is explicit, visible, and operator-authenticated
- operators can distinguish pending, approved, denied, and aborted draft-review states
- approved drafts can be handed off as explicit runtime artifacts for manual follow-through
- no outward email action is claimed or implied without a separate explicit support boundary

### Gate 5: The Supported Use Case Is Believable

Operators can credibly use `v1.8` for:

- reviewing a single selected email or thread through a bounded intake payload
- producing explicit triage guidance and an optional draft reply
- requesting and inspecting approval for a draft suggestion
- materializing an approved draft handoff artifact for manual follow-through
- using the existing control-plane runtime to inspect the resulting state

### Gate 6: Scope Remains Narrow

The release must stay within the declared `v1.8` support boundary:

- careful email triage, not autonomous mailbox management
- one explicit external adapter, not many-channel sprawl
- visible runtime state and approvals, not hidden automation
- supportable operator flow, not “magic inbox autopilot”

If a candidate depends on sending email, mutating mailboxes, or broad channel expansion, it belongs after `v1.8`.

## Done Means Careful External Triage, Not Inbox Automation

For `v1.8`, “done” does not mean ClarityClaw becomes a full email client or autonomous inbox agent.

It means:

- one bounded email intake path is explicit and supportable
- triage output is structured and inspectable
- draft-reply review uses visible approvals
- the first external integration strengthens ClarityClaw’s inspectable-runtime identity rather than weakening it
