# ClarityOS v1.4 Checklist

Status: complete

Goal:
Strengthen the assistant's explicit tool layer so it can inspect, retrieve, and act more usefully without losing ClarityOS's inspectability and policy boundaries.

## Slice 1: Tool Registry Structure

Status: completed

Focus:

- make the growing tool layer easier to extend and reason about
- reduce monolithic tool-file pressure without introducing hidden behavior

Candidate work:

- split tool handlers by domain such as repo, runtime, memory, and web
- keep a single explicit registry surface
- tighten tool metadata for capability, path, and domain handling

Completed work:

- split the monolithic tool implementation into explicit utility, repo, web, runtime-inspection, memory, and shared-support modules
- preserved `runtime.tools` as the stable registry facade used by the runtime and tests
- kept tool capability metadata, path/domain handling, and call semantics explicit at the registry boundary

Acceptance cues:

- the tool layer is easier to extend without becoming messy
- policy and agent allowlisting remain explicit and visible

## Slice 2: Retrieval Capability Depth

Status: completed

Focus:

- improve the quality and usefulness of safe retrieval tools
- support more real assistant research and operator tasks

Candidate work:

- deepen repo and runtime inspection tools
- refine `fetch_url` and related bounded retrieval behavior
- improve tool output shapes for downstream answer quality

Current progress:

- added a bounded `list_directory` repo-navigation tool on top of the existing safe file-read posture
- enriched repo retrieval outputs with previews, counts, and matched-file summaries
- enriched web and runtime inspection outputs with compact summaries that are easier for the assistant to reuse
- documented clearer retrieval examples and output shapes so the stronger tool layer is easier to test and interpret

Acceptance cues:

- the assistant can inspect repo and runtime state more usefully
- retrieval outputs are easier to transform into grounded answers

## Slice 3: Safer Action Tools

Status: completed

Focus:

- add narrow, supportable action capability without broad unsafe execution

Candidate work:

- evaluate scoped write or patch tools
- consider narrow operator-safe action helpers
- avoid broad exec-by-default behavior

Current progress:

- added a dedicated `runtime_write` capability for narrow maintenance actions instead of reusing broad file or exec permissions
- added a `maintenance_operator` agent profile for explicit session, queue, worker, and workflow maintenance helpers
- implemented the first action helpers as operator-safe wrappers around existing runtime operations such as session archival, queue promotion/repair, worker repair, and workflow recovery/resume flows
- added focused denial coverage so non-maintenance agents do not quietly gain access to maintenance actions
- explicitly decided that `v1.4` stays runtime-maintenance-only on the action side; narrow repo write helpers are deferred beyond this milestone
- documented concrete maintenance examples and denial examples so the action-tool boundary is easier to test and operate

Acceptance cues:

- ClarityOS becomes more useful for real tasks without losing explicit safety boundaries
- the shipped action set stays narrow, explicit, and clearly separated from general assistant behavior

## Slice 4: Release Path For Tool Maturity

Status: complete

Focus:

- define what tool-maturity improvement means for `v1.4`

Candidate work:

- document the narrow supported use cases improved by `v1.4`
- define validation and release gates for the stronger tool layer
- define what remains deliberately out of scope

Current progress:

- added an explicit `v1.4` release path document that defines the first supported tool-maturity use case
- defined release gates for registry structure, retrieval usefulness, narrow maintenance actions, and scope control
- clarified that `v1.4` improves tool maturity and bounded operator maintenance, not broad mutation power

Acceptance cues:

- `v1.4` can be described as tool-maturity progress rather than just more tools
- release status can be justified by stronger bounded retrieval, narrow maintenance actions, and explicit scope control
