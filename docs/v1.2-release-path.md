# v1.2 Release Path

This document defines the first narrow deployed assistant use case for ClarityClaw `v1.2` and the release gates required to call that use case supportable.

## First Deployment Use Case

ClarityClaw `v1.2` is aimed at a narrow assistant deployment:

- single-tenant
- self-hosted
- browser-first
- session-backed
- operator-managed
- explicitly authenticated

In concrete terms, the first supported `v1.2` deployment path is:

`A self-hosted internal assistant surface, served through /assistant and optionally /widget, where session ownership, embed posture, and operator maintenance are explicit and bounded.`

## Supported Shape

The `v1.2` release path supports:

- the web-first assistant at `/assistant`
- the embeddable widget at `/widget` and `/widget.js`
- explicit session ownership via per-session tokens
- explicit operator access via `X-Operator-Token`
- widget deployment controls for:
  - allowed origins
  - allowed agents
  - branding
  - launcher posture
  - surface enable/disable
- operator maintenance through:
  - session control views
  - session archive
  - session prune
  - workflow, queue, and worker recovery

## Deliberately Out Of Scope

The `v1.2` release path does not claim support for:

- Telegram, Slack, Discord, email, SMS, or other non-browser channels
- hosted multi-tenant operation
- anonymous public assistant deployment
- broad arbitrary agent access through the widget
- plugin marketplace behavior
- consumer-grade account systems or support tooling

`v1.2` is the assistant deployment hardening release, not the broad assistant-platform release.

## Release Gates

ClarityClaw `v1.2` is ready for this deployment path only if all of the following are true.

### Gate 1: Validation Passes

- `scripts/run_release_validation.sh` passes cleanly.
- `python -m unittest discover -s tests -v` passes cleanly.

### Gate 2: Assistant Surface Auth Is Explicit

- assistant-facing sessions require `X-Session-Token`
- operator routes require `X-Operator-Token` when configured
- no deployed assistant flow depends on implicit shared session ids alone

### Gate 3: Widget Posture Is Explicit

- widget enable/disable is chosen deliberately
- allowed origins are configured intentionally
- allowed agents are configured intentionally
- launcher defaults and branding are configured through env vars rather than code edits

### Gate 4: Operator Playbooks Are Runnable

Operators can run the documented playbooks for:

- assistant-surface incidents
- widget deployment mistakes
- session cleanup
- queue cleanup
- worker repair
- workflow recovery

### Gate 5: Maintenance Flows Are Safe

Operators can safely:

- inspect a session through `GET /sessions/{session_id}/control`
- archive a session through `POST /sessions/{session_id}/archive`
- prune old archived session state through `POST /sessions/prune`
- recover the underlying workflow when a session incident is actually runtime state

### Gate 6: Scope Remains Narrow

The release must stay within the declared `v1.2` support boundary:

- browser-first, not many-channel
- self-hosted, not hosted SaaS
- single-tenant, not general multi-tenant
- operator-managed, not broadly self-service
- bounded assistant surface, not broad platform sprawl

If a candidate depends on wider channel reach or broader product expectations, it belongs after `v1.2`.

## Done Means Deployable

For `v1.2`, “done” does not mean ClarityClaw matches OpenClaw’s full surface area.

It means:

- the assistant surfaces have explicit ownership and deployment posture
- embeds are configurable and fail closed
- operators have concrete maintenance and recovery procedures
- one narrow browser-first assistant deployment path is believable and supportable

That is the line between “assistant demo” and “deployable assistant profile.”
