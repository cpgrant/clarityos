# ClarityClaw v1.2 Snapshot

Milestone snapshot for `v1.2`, the assistant deployment hardening release.

## Status

- Release: `v1.2`
- Focus after this release: `v1.3` planning
- Next target after this release: define `v1.3` scope from release experience

## What v1.2 Added

- session ownership and `X-Session-Token` auth for assistant-facing browser surfaces
- an explicit widget deployment posture around enable/disable state, allowed origins, allowed agents, launcher defaults, and branding controls
- operator playbooks for assistant-surface incidents, widget deployment mistakes, and session cleanup
- session archive and prune maintenance flows on top of the existing recovery-oriented control plane
- a documented first deployed assistant path with explicit release gates

## Acceptance Criteria Met

- Assistant-facing sessions have explicit ownership and token-backed access rather than relying on shared ids alone.
- Widget deployment posture is explicit, configurable, and fail-closed.
- Operators have concrete assistant-surface playbooks and safe maintenance flows.
- One narrow browser-first deployed assistant profile is documented and supportable before broader channel expansion.

## Supported Shape

`v1.2` supports a narrow, self-hosted assistant deployment shape:

- browser-first assistant at `/assistant`
- optional embeddable widget through `/widget` and `/widget.js`
- explicit session ownership and operator-authenticated maintenance
- session archive/prune, workflow recovery, queue repair, and worker repair available to operators

It deliberately does not yet include:

- Telegram
- Slack
- Discord
- hosted hub behavior
- broad multi-channel routing
- multi-tenant browser delivery
