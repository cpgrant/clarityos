# ClarityClaw v0.9 Snapshot

Milestone snapshot for `v0.9`, the production-hardening release.

## Status

- Release: `v0.9`
- Focus after this release: `v1.0` release readiness
- Next target after this release: `v1.0` slice 1 soak, load, and recovery validation

## What v0.9 Added

- versioned persisted state envelopes with migration and inspection endpoints
- workflow, job, worker, memory, approval, and artifact migration support
- operator recovery, replay, safe-resume, prune, repair, and reset actions
- incident-focused observability with stable correlation IDs, structured timelines, and compact summary views
- operator auth guards and production-aware policy validation
- restart, partial-failure, and persisted recovery validation coverage

## Acceptance Criteria Met

- Durable state formats are versioned and have a clear migration path.
- Operators can inspect, replay, prune, and recover workflows, jobs, artifacts, and memory safely.
- Auth, access control, retention, and audit expectations are defined for production deployment.
- Queue, worker, memory, and workflow subsystems expose enough observability for incident debugging.
- System behavior under retry, reclaim, restart, and partial-failure conditions is documented and tested.
- Production hardening includes load, soak, and recovery validation beyond the minimal unit suite.

## Key Runtime Notes

- Persisted runtime state now writes through a shared `{schema, version, payload}` envelope and still reads legacy pre-`v0.9` raw JSON state.
- The control plane can correlate workflow, job, worker, and trace history into a unified incident causality chain.
- Operator endpoints can be protected with `CLARITYOS_OPERATOR_TOKEN`, and production policy posture is activated with `CLARITYOS_ENV=production`.
- The canonical up-to-date documentation remains the repository root `README.md`.
