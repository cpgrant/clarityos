# ClarityOS v1.1 Snapshot

Milestone snapshot for `v1.1`, the first assistant surface and session gateway release.

## Status

- Release: `v1.1`
- Focus after this release: `v1.2` assistant deployment hardening
- Next target after this release: `v1.2` slice 1 session ownership and surface auth

## What v1.1 Added

- explicit persisted sessions with conversation-to-workflow mapping and session control summaries
- a web-first assistant surface backed by the existing `/sessions` flow
- an operator console with session activity, workflow and incident inspection, queue health, worker health, and recovery actions
- a thin embeddable web widget gateway through `/widget`, `/widget.js`, and `/widget/config`
- widget origin controls and branding defaults so embed posture stays explicit and configurable

## Acceptance Criteria Met

- The runtime supports explicit conversation/session records that map inbound user interactions to workflows, memory scope, and operator-visible history.
- A first assistant surface exists, web-first, and still uses the existing runtime instead of introducing a second execution path.
- Operators can inspect sessions, related workflows, incidents, and recovery actions through a simple UI rather than API-only access.
- The first gateway adapter remains thin and transport-focused, with workflow, queue, memory, and recovery behavior still owned by the runtime core.

## Supported Shape

`v1.1` supports a narrow, self-hosted assistant-system shape:

- web assistant at `/assistant`
- operator console at `/operator`
- embeddable widget gateway at `/widget` and `/widget.js`
- persisted sessions, workflows, and typed memory underneath

It deliberately does not yet include:

- Telegram
- Slack
- broad multi-channel routing
- hosted relays or multi-tenant delivery
- plugin marketplace behavior
