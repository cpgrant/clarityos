# v1.3 Release Path

This document defines the narrow assistant-quality release path for ClarityClaw `v1.3` and the release gates required to call that path measurably better than `v1.2`.

## First Quality-Focused Use Case

ClarityClaw `v1.3` is aimed at a narrow but more useful assistant shape:

- single-tenant
- self-hosted
- browser-first
- session-backed
- operator-visible
- repo- and runtime-aware

In concrete terms, the first supported `v1.3` quality path is:

`A self-hosted browser assistant for repo, roadmap, runtime, and operator questions, served through /assistant and optionally /widget, with grounded answer shaping and operator-visible session/workflow inspection.`

## Supported Shape

The `v1.3` release path supports:

- the web-first assistant at `/assistant`
- the embeddable widget at `/widget` and `/widget.js`
- the operator console at `/operator`
- grounded assistant answers for:
  - repo and docs questions
  - milestone and roadmap planning questions
  - bounded comparisons against controlled external references
  - runtime questions that include explicit session or workflow identifiers
- guarded tool-guided grounding through:
  - repo search and file excerpts
  - compact runtime inspection summaries
  - controlled external fetch domains
- browser-surface validation through the smoke suite for assistant, operator, and widget flows

## Deliberately Out Of Scope

The `v1.3` release path does not claim support for:

- broad many-channel expansion
- media-heavy or voice-first assistant behavior
- broad arbitrary web research outside the configured fetch domains
- plugin marketplace behavior
- full autonomous coding or unrestricted execution
- hosted multi-tenant operation

`v1.3` is the assistant-quality release, not the broad assistant-platform release.

## Release Gates

ClarityClaw `v1.3` is ready for this quality path only if all of the following are true.

### Gate 1: Validation Passes

- `scripts/run_release_validation.sh` passes cleanly.
- `python -m unittest discover -s tests -v` passes cleanly.
- `python -m unittest tests.test_browser_smoke -v` passes cleanly.

### Gate 2: Grounding Is Explicit

- assistant-surface project questions inject repo context instead of relying on a generic model-only answer
- tool-guided grounding respects the active agent's allowlisted tools and policy
- controlled external fetches stay within the configured domains
- runtime questions with explicit session or workflow IDs can surface compact inspection summaries

### Gate 3: Final Answer Shape Is Deliberate

- comparison questions get a clear current-position, similarities, gaps, and next-step structure
- planning questions get a clear objective, current state, next slices, and out-of-scope structure
- status-style questions get a clear objective, current state, what is working, and what comes next structure
- grounded answers are easier to scan and less repetitive than the `v1.2` baseline

### Gate 4: Browser Surfaces Are Trustworthy

- `/assistant` shows clearer owned-session, waiting, errored, and recovered states
- `/widget` shows clearer host-policy and owned-thread state
- `/operator` shows meaningful loading, empty, and error placeholders instead of blank panes
- browser-surface smoke tests cover the real assistant, operator, and widget route flows

### Gate 5: Supported Use Cases Are Believable

Operators can credibly use `v1.3` for:

- browser-first ClarityClaw repo and roadmap research
- milestone planning and product-direction questions
- runtime troubleshooting with explicit session/workflow identifiers
- bounded comparisons against OpenClaw, CrewAI, and OpenAI docs when those references are within the configured fetch policy

### Gate 6: Scope Remains Narrow

The release must stay within the declared `v1.3` support boundary:

- better assistant quality, not more channel breadth
- stronger grounding, not broader hidden magic
- browser-first use cases, not general assistant-platform sprawl
- bounded retrieval and inspection, not arbitrary unrestricted capability growth

If a candidate depends on broad integrations or a much wider assistant surface, it belongs after `v1.3`.

## Done Means More Useful

For `v1.3`, “done” does not mean ClarityClaw matches OpenClaw’s breadth.

It means:

- the browser assistant is more grounded and less generic
- the assistant uses explicit tools more intelligently for repo, runtime, and bounded comparison questions
- final answers are more structured and easier to scan
- browser surfaces are easier to trust and validate
- one narrow browser-first assistant-quality path is measurably better than `v1.2`

That is the line between “deployable assistant surface” and “more useful assistant system.”
