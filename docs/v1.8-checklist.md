# ClarityClaw v1.8 Checklist

Status: in progress

Goal:
Add one narrow external integration for email triage without turning ClarityClaw into a broad autonomous inbox manager or many-channel assistant product.

Companion docs:

- [`docs/roadmap.md`](./roadmap.md)
- [`docs/differentiators.md`](./differentiators.md)
- [`docs/v1.8-release-path.md`](./v1.8-release-path.md)

## Slice 1: Email Intake Boundary And Runtime Handoff

Status: complete

Implemented so far:

- `POST /intake/email` provides one bounded email intake path into the existing runtime
- email-origin input preserves explicit source metadata such as account, thread, message, timestamps, and sender context
- email intake maps into normal sessions and workflows instead of creating a second hidden execution path
- intake validation enforces narrow payload and content limits

Focus:

- keep the first external integration thin rather than building a full mailbox platform
- preserve ClarityClaw's main shape: thin surfaces over an explicit session, workflow, queue, and operator runtime

Candidate work:

- define one supported intake path for email-oriented input such as webhook delivery, controlled API pull, or selected-message ingestion
- preserve explicit source metadata such as account identity, thread identifier, message identifier, timestamps, and sender context
- map external email intake into the existing session and workflow runtime rather than adding a second hidden execution path
- define narrow ingestion limits for batch size, message size, and supported email content shape

Acceptance cues:

- operators can explain exactly how email enters the runtime and where that state is visible afterward
- the first email integration feels like a bounded adapter into ClarityClaw, not a new mailbox subsystem

## Slice 2: Structured Email Triage And Draft Assistance

Status: complete

Implemented so far:

- email triage output is parsed into structured fields such as bottom line, urgency, bucket, next action, and draft reply
- successful triage runs persist a first-class `email_triage` artifact linked to the workflow
- intake responses expose structured triage output for downstream use
- operator-facing workflow/session views now surface structured email triage instead of only a raw artifact record
- the current `v1.8` path supports a real thread-review and draft-assistance flow rather than only raw prompt output

Focus:

- make one real email use case useful soon without taking on full inbox autonomy
- keep output explicit, inspectable, and easy to approve or reject

Candidate work:

- support structured triage outputs such as summary, urgency, suggested bucket, follow-up recommendation, and draft-reply suggestion
- define one narrow email-tidy-up workflow such as "summarize and sort this inbox slice" or "review this thread and draft a reply"
- keep the first useful path read-heavy or draft-heavy before broad mailbox mutation is introduced
- expose a stable output shape that can be consumed by assistant, widget, or future external surfaces

Acceptance cues:

- a user can hand ClarityClaw a small inbox batch or one thread and get a useful structured triage result
- the system is helpful for email cleanup without yet claiming autonomous inbox management

## Slice 3: Consent, Approvals, And Operator Visibility For Email Work

Status: complete

Implemented so far:

- draft replies from `email_triage` artifacts can request an explicit approval record before any future outward email action
- workflow and operator views surface email-triage approval state and whether draft approval is still pending or not yet requested
- operator workflow actions expose a bounded `Request Draft Approval` path tied to the triage artifact rather than a hidden mailbox mutation path
- post-approval outcomes now spell out what approved, denied, aborted, and pending draft states mean for the operator in-product
- approved draft replies can now materialize a first-class `email_draft_handoff` artifact for manual follow-through without implying automatic send behavior
- operator-authenticated API paths now exist for both draft approval requests and approved-draft handoff creation

Focus:

- preserve trust as ClarityClaw touches more personal and operationally sensitive input
- keep external-use behavior aligned with explicit memory, visible runtime state, and operator supportability

Candidate work:

- require explicit approval boundaries before any mailbox-changing or message-sending action
- expose email-origin sessions, workflows, and action recommendations clearly in operator views
- keep preference or user-specific behavior explicit rather than hidden inside prompt state
- document how email-derived context, summaries, or memories are retained, reviewed, and deleted

Acceptance cues:

- operators and users can tell what the assistant saw, what it inferred, and what still requires approval
- email support strengthens ClarityClaw's inspectable-runtime identity instead of weakening it

## Slice 4: Release Path For Narrow Email Triage

Status: in progress

Implemented so far:

- a first `v1.8` release-path document now defines the narrow supported email-triage shape, its release gates, and what remains out of scope
- release validation now includes a bounded intake-to-triage-to-draft-approval-to-handoff drill using synthetic email data
- the current suite validates the email-triage path across runtime, control-plane, API, and browser-smoke coverage
- the release-path doc now includes a concrete operator walkthrough for intake -> triage -> approval -> approved handoff

Focus:

- define one believable `v1.8` release path around careful email triage integration
- keep scope narrow enough that the release remains supportable and testable

Candidate work:

- document the first supported email-triage use case and deployment posture
- define release gates for ingestion clarity, structured triage usefulness, approval safety, and operator visibility
- define what remains out of scope, especially autonomous mailbox management and many-channel expansion
- add validation coverage using bounded synthetic email fixtures and end-to-end intake-to-triage checks

Acceptance cues:

- `v1.8` can be described as careful external integration for narrow email triage rather than generic channel expansion
- release readiness can be justified by explicit boundaries, useful triage behavior, and supportable operator visibility
